# Skill: write

## 目的
`knowledge/concepts/frameworks/note執筆ガイド.md` に定義されたスタイルガイドに従い、note.com向けの記事を執筆する。

## トリガー
ユーザーが「記事を書いて」「ドラフトを作って」「執筆して」「本文を書いて」などと指示したとき。

## 手順

### 1. ガイドと参照知識の読み込み
- **必ず** `knowledge/concepts/frameworks/note執筆ガイド.md` を読む
- 記事ファイルが既にある場合は `📋 執筆メモ` からモード・スタンス・ターゲットを把握する
- `🔗 参照知識` に記載された概念ノートを全て読む
- 連載記事の場合は `knowledge/concepts/series/` 内の該当設計書も読む
- `stances/` 内の関連スタンスを確認し、論旨が矛盾しないことを担保する

### 2. 構成の提案（本文執筆前に必ず行う）
以下の形式で提示し、ユーザーの確認を取る：

```markdown
## 推定
- モード：（トレード / 雑記）
- タイプ：（連載 / 単発）
- スタンス：（バズ寄り / 思考ログ寄り / ハイブリッド）
- 有料ライン：（全文無料 / 〇〇まで無料→有料）

## タイトル案
- 案1：
- 案2：
- 案3：

## 構成案
- 導入：
- 第1章：
- 第2章：
- 第3章：
- まとめ：

## 参照知識候補
（Obsidianで紐づけるべき概念ノートを [[リンク]] 形式で列挙）

## メモ
- スタンスの根拠や補足を一言
```

ユーザーが「すぐ本文を書いて」と言った場合のみ、確認なしで本文に進んでよい。

### 3. 本文執筆
- 執筆ガイドのトーン・表現ルール・禁止事項に厳密に従う
- **ユーザーから指定がない場合、以下のデフォルト設定を適用する：**
  - **語尾： 「だ・である」調（常体）**
  - **分量： 2000〜3000文字程度**
- `stances/` の関連スタンスと矛盾しないか常に意識する
- 連載記事の場合、設計書の「執筆注意点」に違反しないか確認する
- 記事の `📋 執筆メモ` に記載されたターゲット読者に合わせた語彙・説明レベルを維持する

### 4. ドラフト出力フォーマット
本文ドラフトは以下の形式で出力する（Obsidianにそのまま貼り付けられる形）：

```markdown
---
title: 記事タイトル
tags: [タグ1, タグ2]
created: YYYY-MM-DD
status: draft
note_url: 
gist_tables: []
related:
---

## 📋 執筆メモ
- モード：
- タイプ：
- スタンス：
- 有料ライン：
- ターゲット：

## 🔗 参照知識
（[[概念名]] 形式で列挙）

---

## 📝 本文

（本文ドラフト）
```

### 5. 仕上げ
- 執筆ガイドの表現ルール・禁止事項で最終チェックする
- note向けの改行・見出しレベル最適化を施す
- `review` スキルの観点で自己点検する（概念ノートとの整合性）

---

## 記事修正・ブラッシュアップの場合

ユーザーが既存記事の修正を依頼した場合：

1. 記事ファイルを読む
2. `knowledge/concepts/frameworks/note執筆ガイド.md` を読む
3. 参照知識との整合性を確認する
4. 修正箇所を **変更前→変更後** の形式で提示する
5. ユーザー承認後にファイルを直接編集する

**重要**: 論旨・主張の変更は提案→承認フロー必須。勝手に書き換えない。
